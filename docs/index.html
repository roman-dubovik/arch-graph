The problem

Twelve services in. Nobody knows what depends on what.

You're 12 services into a NestJS monorepo. Who publishes user.created? Which service depends on users_table? What breaks if you rename auth-lib? Each question is a 15-minute grep session — or a 30-second LLM hallucination. arch-graph turns the entire monorepo into a typed graph you can query in milliseconds, with every answer citing file:line.

How it works

Sources flow in. A typed graph grows out.

Every @Module, @InjectQueue, @MessagePattern, @InjectRepository, axios.get() and import statement gets parsed by ts-morph, classified by domain extractor, resolved to typed nodes and edges, and emitted as graph.json.

Quick start

Three commands.

  1. 1

    Install

    Clone, run the installer. Symlinks arch-graph into ~/.local/bin. Requires Node ≥ 20.

    git clone https://github.com/roman-dubovik/arch-graph ~/.arch-graph
bash ~/.arch-graph/scripts/install.sh
  2. 2

    Init

    Interactive wizard. Picks defaults, writes arch-graph.config.ts, optionally installs the Claude Code skill and git hook.

    cd path/to/your/nestjs-monorepo
arch-graph init
  3. 3

    Build

    Emits graph.json, diagnostics.json, validation.json, graph.mermaid — plus an advisory recall table per domain.

    arch-graph build               # advisory: always exit 0
arch-graph build --strict      # CI: exit 3 if recall drops below floor

CLI

Query the graph from the terminal.

Ten subcommands. --json by default, --table for humans. Exit code 4 for not-found.

who-publishes

Find every service that publishes on a NATS subject. Cites file:line.

arch-graph who-publishes user.created --table

deps-of

List the outgoing dependencies of a service, grouped by edge kind.

arch-graph deps-of platform-api --table

path

Shortest directed path between two nodes. Useful for "is X reachable from Y?".

arch-graph path service:auth service:billing

stats

Node + edge counts per kind. Sanity-check the graph after every build.

arch-graph stats

Integration

Always-on inside Claude Code.

arch-graph claude install --skill writes a delimited block into ./CLAUDE.md telling Claude to consult the graph before answering architecture questions, and registers ~/.claude/skills/arch-graph/SKILL.md globally. The pre-commit hook keeps the graph in lockstep with the code — every checkout in history is self-consistent. 

arch-graph claude install --skill
arch-graph hook install            # pre-commit, recommended

Honesty

What it isn't.

arch-graph is a static extractor. It does not see runtime configuration, container env values, or dynamically constructed identifiers.

  • Static, not runtime

    If your subject name only exists at runtime (env vars, DI factories), arch-graph won't invent an edge. It records the gap in diagnostics.json.

  • Dynamic subjects → diagnostics

    publish(`subject.${userId}`) becomes a diagnostic with file:line, not a guessed edge. No hallucinated topology.

  • Single monorepo

    No cross-repo graph yet. Multi-repo deployments need a separate stitching pass that isn't built.

  • NestJS-specific

    The extractors are tuned to NestJS decorators and a small set of wrapper conventions. General-purpose code analysis isn't the goal — graphify is.
